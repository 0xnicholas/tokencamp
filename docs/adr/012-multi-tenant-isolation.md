# ADR-012: 多租户隔离模型

**状态**: 已采纳

**日期**: 2026-05-29

**决策者**: Nicholas

## 背景

Tokencamp 作为 SaaS 产品，多个付费客户（Organization）共享同一套基础设施。必须保证：(1) 租户之间数据不可见；(2) 一个租户的流量不能挤占其他租户的资源；(3) 配额和预算严格隔离。

## 决策

采用 **逻辑隔离** 模型（共享数据库，通过 `org_id` 分区），配以多层限流保障资源公平。

### 隔离层级

```
┌─────────────────────────────────────────────────────┐
│ 层 3: 预算隔离                                       │
│ 每个 Org/Team/Key 有独立 budget。调用前 Hook 检查     │
│ total_spend >= max_budget → 429 budget_exceeded     │
├─────────────────────────────────────────────────────┤
│ 层 2: 速率隔离                                       │
│ 每个 Key 有独立的 TPM/RPM 限制。Redis 令牌桶          │
│ 超限 → 429 rate_limit_exceeded                      │
├─────────────────────────────────────────────────────┤
│ 层 1: 模型访问隔离                                    │
│ 每个 Key 有 allowed_models 白名单                    │
│ 不在白名单的模型 → 403 model_not_allowed             │
├─────────────────────────────────────────────────────┤
│ 层 0: 认证隔离                                       │
│ API Key 唯一标识租户身份。key_hash → org_id          │
│ 无效 Key → 401 unauthorized                         │
└─────────────────────────────────────────────────────┘
```

### 为什么是逻辑隔离而非物理隔离

| 维度 | 逻辑隔离 (共享 PG) | 物理隔离 (每租户独立 PG) |
|------|--------------------|--------------------------|
| 数据不可见 | ✅ `WHERE org_id = ?` 保证 | ✅ 天然的 |
| 管理复杂度 | 低，单一 schema | 高，N 个数据库实例 |
| 资源公平 | 需应用层限流 | 各自独立资源池 |
| 成本 | 低 | 高（尤其长尾小租户）|
| 跨租户查询（运营）| 简单 | 需要联邦查询 |

MVP 阶段采用逻辑隔离。当出现需要物理隔离的高端客户时，通过 `org_id` 路由到独立数据库实例即可，应用层代码不变。

### 资源争抢缓解

即使有每 Key 的 TPM/RPM 限制，高流量租户仍可能影响低流量租户（抢占 HTTP 连接池、CPU 等）。缓解措施：

1. **连接池分片**: 按 `api_base`（目标 Provider endpoint）分组 HTTP 连接池，限制每个 destination 的最大连接数。这确保一个 Provider 的故障不会耗尽所有可用连接，同时多租户共享同一 destination 的连接池（避免连接数爆炸）
2. **优先级队列**: 引入租户级别（free/pro/enterprise）的优先级，高优先级请求排前面
3. **全局限流**: Gateway 级别的全局 TPM 上限，触发时对所有租户按比例降级

MVP 不实现优先级队列（过度设计），但连接池分片在 Router 初始化时根据 `model_list` 自动按 `api_base` 分组，天然隔离不同 Provider endpoint，同时限制每个 endpoint 的最大并发连接数。

## 后果

- **正面**: 管理简单，成本低，容易做跨租户分析
- **负面**: 资源隔离不彻底，恶意或失控租户可能影响邻居
- **缓解**: 多层限流 + 监控告警。对于需要严格隔离的客户，通过独立部署实例实现物理隔离

## 备选方案

- **物理隔离（每租户独立 PG）**: 最强的隔离保证，但运维成本和资源浪费严重。适合 10 个以下大客户，不适合数百个小租户
- **Hybrid 隔离**: premium 租户独立 PG，free 租户共享 PG。Phase 3 考虑
