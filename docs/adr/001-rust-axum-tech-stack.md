# ADR-001: Rust + Axum 作为技术栈

**状态**: 已采纳

**日期**: 2026-05-30

**决策者**: Nicholas

## 背景

Tokencamp 需要构建一个 LLM API Gateway，核心功能包括：统一接入多模型提供商、智能调度、多租户管理。需要选择一个高性能、内存安全、适合构建基础设施层服务的技术栈。

## 决策

选择 **Rust 1.85+** 作为编程语言，**Axum** 作为 Web 框架，**Tokio** 作为异步运行时。

## 理由

1. **极致性能与可预测延迟**: Rust 的零成本抽象和无 GC 保证请求路径上没有不可预测的暂停。LLM Gateway 是 IO 密集型服务（大量并发 HTTP 调用），Tokio 异步运行时可以高效调度数十万并发连接，吞吐量远超 Python asyncio。

2. **真正的多线程并发**: Rust 没有 GIL，可以利用所有 CPU 核心。Gateway 的请求处理、Token 计数、加密解密、序列化等 CPU 工作可以真正并行，不互相阻塞。

3. **编译期类型安全**: Rust 的类型系统和所有权模型在编译期消除空指针异常、数据竞争、use-after-free 等整类运行时错误。对于基础设施层服务，编译期保证远比运行时测试可靠。

4. **低资源占用**: Rust 服务的内存占用通常是 Python 的 1/5 - 1/10，冷启动时间在毫秒级（vs Python 的秒级）。在 K8s 环境下，更小的资源请求意味着更高的装箱密度和更快的自动扩缩容。

5. **成熟的异步生态**: `reqwest`（HTTP 客户端）、`sqlx`（异步数据库驱动）、`redis-rs`（异步 Redis）、`serde`（序列化）等核心库经过大规模生产验证。`tower` 中间件生态提供了限流、超时、重试、负载均衡等开箱即用的组件。

6. **长期维护优势**: Rust 的「Fearless Refactoring」特性让大型代码库的重构变得安全可控。强类型系统使新开发者能在编译器引导下理解代码，降低长期维护成本。

## 后果

- **正面**: 高性能、低延迟、低资源占用、编译期安全保证
- **负面**: 开发速度低于 Python，迭代周期较长（编译时间 + 更严格的类型建模）
- **负面**: 团队学习曲线陡峭（所有权、生命周期、异步模型）
- **负面**: LLM SDK 生态不如 Python 丰富（`async-openai` 等 crate 可用但不如 Python `openai` 库成熟），部分 Provider 需要手动实现 HTTP 适配
- **缓解**: 核心 LLM 调用逻辑（协议转换、流式处理）通过 `reqwest` 自行实现，成本可控。Provider 适配器接口定义清晰（见 ADR-003），新增 Provider 只需实现 trait

## 备选方案

- **Python (FastAPI)**: 原方案，LLM 生态最丰富，开发速度最快。但 GIL 瓶颈、内存占用、冷启动时间在高并发 SaaS 场景下是硬伤。LiteLLM 用 gunicorn 多进程缓解，但进程间无法共享连接池和缓存，资源浪费严重。
- **Go (Gin/Fiber)**: 高性能，goroutine 并发模型优秀。但 LLM SDK 生态弱，每个 Provider 需要自己封装 HTTP 调用。Go 的零值模型和错误处理在复杂业务逻辑中容易产生隐蔽 bug。
- **Node.js + TypeScript**: 前后端统一语言，但单线程事件循环在高 CPU 操作（Token 计数、加密）时会阻塞所有请求。
