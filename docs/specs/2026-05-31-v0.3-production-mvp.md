## 1. 目标

将 Tokencamp 从「可运行的原型」提升至「可投入生产的 MVP」：

- **PostgreSQL 集成**：api_keys + spend_logs 表，API Key 权威源从 YAML 迁移到数据库
- **Redis DualCache**：in-memory LRU + Redis 双层缓存，加速 Key 查询和限流
- **Hook 系统**：ProxyHook trait + YAML 注册，解耦横切关注点
- **TPM/RPM 限流**：`ParallelRequestLimiter` Hook，Redis INCR + TTL
- **成本追踪**：`CostTracker` Hook，Redis queue + 定时批量 Flush PG
- **管理 API**：`/admin/keys/generate` 创建 Key，`/admin/keys` 列出
- **健康检查**：主动探测 + Router 配合过滤
- **Fallback 降级**：Cooldown/Retry 耗尽后自动切换到备选模型

### 不做

- Prometheus / OTel（v0.4）
- 配置热更新（v0.4）
- 密钥加密存储（v1.0）
- Embeddings 端点（v0.4）

    completion_tokens INTEGER NOT NULL DEFAULT 0,

);

CREATE INDEX idx_spend_logs_key ON spend_logs(key_id);

CREATE INDEX idx_spend_logs_request ON spend_logs(request_id);

### 2.3 API Key 认证流

```
v0.2 (旧):  YAML api_keys 列表匹配
v0.3 (新):  SHA-256(request_key) → DualCache.get(key_hash)
              ├── hit → UserAPIKeyAuth ✓
              └── miss → SQLx SELECT BY key_hash → hydrate cache → ✓ / 401
```

**Key 创建**：`POST /admin/keys/generate`（需 master_key）→ `sk-tc-{32 random hex}`，SHA-256 存 PG，返回明文（仅此一次）。

**master_key**：保留 YAML 配置，用于 admin API 认证。不做 DualCache。

## 3. DualCache

### 3.1 结构

```rust
pub trait CacheLayer: Send + Sync {
    async fn get(&self, key: &str) -> Option<String>;
    async fn set(&self, key: &str, value: &str, ttl: Duration);
    async fn del(&self, key: &str);
}

pub struct DualCache {
    in_memory: LruCache<String, CacheEntry>,
    redis: Option<MultiplexedConnection>,
}
```

### 3.2 读写路径

```
读: in-memory → miss → Redis → miss → PostgreSQL
写: in-memory + Redis pipeline（不阻塞）
```

### 3.3 缓存 Key 设计

| 用途 | Key 格式 | TTL |
|------|---------|-----|
| API Key 查询 | `key:{hash}` | 5min |
| TPM 计数器 | `key:{key_id}:tpm:{window}` | 60s |
| RPM 计数器 | `key:{key_id}:rpm:{window}` | 60s |
| Cooldown | `deployment:{id}:cooldown` | 配置 |
| 失败计数器 | `deployment:{id}:failures` | 120s |
| Spend 队列 | `spend:queue` | 持久 |
| Spend 锁 | `spend:lock` | 60s（刷新） |

### 3.4 无 Redis 降级

Redis URL 未配时：DualCache 退化纯 in-memory LRU。限流计数退化单实例内存计数。spend 队列退化直接同步写 PG（性能下降但可用）。

## 4. Hook 系统

### 4.1 Trait

```rust
#[async_trait]
pub trait ProxyHook: Send + Sync {
    async fn async_pre_call_hook(
        &self, request: &ChatRequest, auth: &AuthContext,
    ) -> Result<(), HookError>;

    async fn async_post_call_hook(
        &self, request: &ChatRequest, response: &ModelResponse, auth: &AuthContext,
    );

    async fn async_on_error_hook(
        &self, request: &ChatRequest, error: &ProviderError, auth: &AuthContext,
    );
}

pub struct AuthContext {
    pub key_id: String,
    pub key_name: Option<String>,
    pub tpm_limit: Option<u32>,
    pub rpm_limit: Option<u32>,
}
```

### 4.2 注册

```yaml
# config/default.yaml
hooks:
  enabled:
    - parallel_request_limiter
    - cost_tracker
```

Gateway 启动时按配置顺序构建 `Vec<Box<dyn ProxyHook>>`。

### 4.3 执行顺序

```
pre_call hooks（顺序执行，任一失败 → 阻断 + 返回错误）
  → 路由 + Provider 调用
  → post_call hooks（tokio::spawn 异步，不阻塞响应）
  → 返回

on_error hooks（在调用失败时触发）
```

如果 Provider 调用成功但 post_call hook 失败——不影响已返回的响应，仅记录日志。

## 5. 限流 Hook（ParallelRequestLimiter）

### 5.1 逻辑

```rust
impl ProxyHook for ParallelRequestLimiter {
    async fn async_pre_call_hook(&self, request, auth) -> Result<(), HookError> {
        let tpm_key = format!("key:{}:tpm:{}", auth.key_id, current_window());
        let rpm_key = format!("key:{}:rpm:{}", auth.key_id, current_window());

        let current_tpm: u32 = cache.incr(&tpm_key).await?;
        let current_rpm: u32 = cache.incr(&rpm_key).await?;
        cache.expire(&tpm_key, 60).await;
        cache.expire(&rpm_key, 60).await;

        if let Some(limit) = auth.tpm_limit {
            if current_tpm > limit { return Err(HookError::RateLimitExceeded("tpm")); }
        }
        if let Some(limit) = auth.rpm_limit {
            if current_rpm > limit { return Err(HookError::RateLimitExceeded("rpm")); }
        }
        Ok(())
    }
}
```

### 5.2 窗口计算

```rust
fn current_window() -> u64 {
    std::time::SystemTime::now()
        .duration_since(std::time::UNIX_EPOCH)
        .unwrap()
        .as_secs() / 60 * 60
}
```

## 6. 成本追踪 Hook（CostTracker）

### 6.1 成本计算

```rust
fn calculate_cost(response: &ModelResponse, model_info: &ModelPricing) -> f64 {
    (response.usage.prompt_tokens as f64 * model_info.prompt_price
     + response.usage.completion_tokens as f64 * model_info.completion_price)
    / 1000.0
}
```

ModelPricing 从配置文件 `model_list[].litellm_params.model_info` 获取。

### 6.2 队列写入

```rust
impl ProxyHook for CostTracker {
    async fn async_post_call_hook(&self, request, response, auth) {
        let entry = SpendEntry {
            key_id: auth.key_id.clone(),
            request_id: response.id.clone(),
            model: request.model.clone(),
            provider: "...".into(),
            prompt_tokens: response.usage.prompt_tokens,
            completion_tokens: response.usage.completion_tokens,
            cost: calculate_cost(response, &model_pricing),
            duration_ms: None,
            status: "success".into(),
        };
        let json = serde_json::to_string(&entry).unwrap();
        let _ = cache.rpush("spend:queue", &json).await;
    }
}
```

### 6.3 批量 Flush（定时任务）

```
每 60s:
  1. 尝试获取分布式锁 spend:lock（SET NX EX 55）
  2. LRANGE spend:queue 0 999 → entries
  3. LTRIM spend:queue 1000 -1 (原子操作前确保不丢)
  4. 批量 INSERT INTO spend_logs (ON CONFLICT request_id DO NOTHING)
  5. 批量 UPDATE api_keys SET total_spend = total_spend + delta
  6. 释放锁
```

无 Redis 时降级为同步写 PG。

### 6.4 读时合并

用量查询 API 同时查 PG + Redis 未 flush 的增量，返回近乎实时的数据。

## 7. 管理 API

### 7.1 端点

```
POST /admin/keys/generate     # 生成新 Key（需 master_key）
GET  /admin/keys               # 列出所有 Key
DELETE /admin/keys/{id}        # 停用 Key (is_active = false)
```

### 7.2 master_key 认证

从 `Authorization: Bearer {master_key}` 提取，与 YAML 配置的 `master_key` 比对。不使用 DualCache。

```yaml
general_settings:
  master_key: ${TOKENCAMP_MASTER_KEY}
```

### 7.3 Key 生成

```rust
async fn generate_key(pool: &PgPool) -> Result<GeneratedKey> {
    let raw = format!("sk-tc-{}", hex::encode(random_32_bytes()));
    let hash = sha2::Sha256::digest(raw.as_bytes());
    let prefix = &raw[..15]; // "sk-tc-" + first 8 hex

    sqlx::query!("INSERT INTO api_keys (key_hash, key_prefix) VALUES ($1, $2)", hash, prefix)
        .execute(pool).await?;

    Ok(GeneratedKey { key: raw, prefix: prefix.to_string() })
}
```

## 8. 健康检查 + Fallback

### 8.1 健康探测

```
后台任务 (tokio::spawn, 每 30s):
  for deployment in active_deployments:
    ├── cooldown 中 → 跳过
    ├── HTTP HEAD / → Provider
    ├── 超时 5s → unhealthy
    ├── 5xx → unhealthy
    └── 成功 → healthy, 记录 latency_ms

Redis: deployment:{id}:health → JSON {status, last_check, latency_ms} (TTL 90s)
```

懒执行：未调用的不探测，10min 无调用移出。

### 8.2 Router 集成

```rust
fn select_deployment() {
    candidates = filter_cooldown(all);
    candidates = filter_unhealthy(candidates);   // 可选配置
    return simple_shuffle(candidates);
}
```

### 8.3 Fallback

```yaml
router_settings:
  fallbacks:
    claude-sonnet-4-5: ["gpt-5"]
```

当 `select_deployment` 返回空时，遍历 fallback 链重新选择。

## 12. 成功标准

| 标准 | 指标 |
|------|------|
| Key 生成 | `POST /admin/keys/generate` → 返回明文 Key |
| Key 认证 | 生成 Key → chat 调用 200，无效 Key 401 |
| 限流 | TPM 超限 → 429，窗口过期恢复 |
| 成本追踪 | chat 调用 → spend_log 写入 PG（延迟 < 60s） |
| 健康检查 | Provider 宕机 → unhealthy → Router 排除 |
| Fallback | 主模型不可用 → 自动切换备选模型 |
| 无 Redis | 降级 in-memory，Chat 功能仍可用 |
| 测试 | 10+ 单元测试通过 |
